# 站点管理页折叠态主行改版设计

## 背景

当前站点管理页的折叠态主行存在以下核心问题：

1. 默认窗口宽度下出现横向滚动，列表信息承载超出可用空间。
2. 折叠态主行同时混合表格式列对齐和卡片式内容堆叠，阅读和比较效率都偏低。
3. 站点名、余额等高优先级信息经常被截断，而右侧动作与低频信息占用了过多空间。
4. 顶部工具区层级混乱，搜索、分组、Header 主操作的归属关系不清楚。
5. 当前主行存在重复的零值说明文本，噪音较高。

本次设计只处理折叠态主行和顶部工具区，不重做展开区承载方式。

## 目标

在保留现有展开区架构和业务逻辑的前提下，重做站点管理页折叠态主行，使其在常规桌面宽度下无横向滚动、信息优先级清晰、站点识别更快、右侧操作更稳定。

## 非目标

以下内容不在本次设计范围内：

1. 不重做 `SiteCardDetails` 展开区内容。
2. 不改展开/收起状态模型，不改 `AppCard.expandContent` 承载方式。
3. 不改后端字段、后端接口或持久化结构。
4. 不改排序语义之外的业务逻辑，不顺带调整 CLI/站点编辑流程。
5. 不删除 `LDC比例` 的后端代码，只在前端隐藏。

## 已确认的总体方向

### 1. 主体承载方式

1. 保留当前“每个站点一张 `SiteCard`，展开区挂在卡片内部”的架构。
2. 只重做折叠态主行，不把列表改造成原生 `<table>`。
3. 折叠态主行继续走“尽量保留更多列同时可见”的方向，而不是大幅压缩为少量聚合列。

### 2. 顶部工具区

1. `站点分组` 保持现有位置与结构，不改为纵向侧栏，也不单独拆到另一层。
2. 搜索框放在 `站点分组` 这一层的最右侧。
3. 页面 Header 最右侧只保留 `添加站点` 和 `恢复` 两个页面级主操作。
4. Header 与分组层的职责拆分如下：
   - Header：页面主操作。
   - 分组层：站点分组切换与站内搜索。

### 3. 表头

1. 表头继续固定在列表顶部，不跟随站点内容滚动。
2. 表头列名调整为：
   - `站点`
   - `余额`
   - `今日消费`
   - `Token统计`
   - `请求统计`
   - `模型数`
   - `CLI可用性`
3. `更新时间` 不再作为独立列。
4. `LDC比例` 列仅前端隐藏，后端相关字段和逻辑不动。
5. 表头整体紧凑化，去掉无意义占位，保证字段名清楚且与主行对齐稳定。

## 折叠态主行设计

### 1. 站点列

站点列改为两行结构：

1. 第一行只显示站点名。
2. 第二行左侧只显示账户名。
3. 第二行右侧显示更新时间。

说明：

1. 更新时间从独立列移入站点列第二行右侧，以回收一列横向空间。
2. 本次不增加额外账户标识，不在第一行追加账户信息。
3. 当同名站点存在多条记录时，依靠第二行账户名区分。

### 2. 余额列

余额列采用按数值阈值切换的混合显示规则：

1. 小于 `100000` 的金额显示完整值。
2. 大于等于 `100000` 的金额显示缩写值。

设计目的：

1. 在控制列宽的同时避免中小金额失去精度。
2. 优先解决“大金额把整列撑爆”与“小金额被过度简化”这两个相反问题。

### 3. 今日消费列

本次不改变“今日消费”的业务语义和跨天归零逻辑，只要求它在新版主行中继续保持可见，并随整体列宽重排后稳定显示。

### 4. Token统计列

1. 保持两层语义：
   - 第一行：总 Token。
   - 第二行：输入 / 输出。
2. 当总 Token 为 `0` 时，只显示一个简短的 `0`，不再显示第二行说明。
3. 排序行为不变，仍只按总 Token 排序。

### 5. 请求统计列

1. 保持两层语义：
   - 第一行：今日请求数。
   - 第二行：RPM / TPM。
2. 当请求主值为 `0` 时，只显示一个简短的 `0`，不再显示第二行说明。
3. 该列继续不支持排序。

### 6. 模型数列

模型数继续保留为独立列，不并入其他指标区域。

### 7. CLI可用性列

1. 原兼容性列改名为 `CLI可用性`。
2. 仍保留现有图标表达方式。
3. 本次只允许缩小图标尺寸、优化间距与列宽，不改变其语义表达模型。

## 右侧操作区设计

### 1. 总体原则

1. 保留现有一排图标动作的基本结构。
2. 主要做压缩、对齐、间距优化，不将它们大规模下沉为纯菜单。
3. 继续保留右侧 `...` 更多菜单，站点主配置入口仍在其中的 `编辑站点`。

### 2. CLI 配置入口

1. 当前纯齿轮图标入口改为显性文字入口 `CLI配置`。
2. `CLI配置` 入口继续承担 CLI 配置相关职责，不承担站点主配置职责。
3. 其余动作继续保持图标形式。

### 3. 站点主配置入口

1. 站点主配置入口仍然在右侧 `...` 更多菜单中的 `编辑站点`。
2. 本次不把站点主配置从更多菜单提升为新的显性文字入口。

## 密度与视觉层级

### 1. 行高策略

1. 折叠态主行使用偏紧凑的行高。
2. 副信息使用更小字号，并尽量贴近主值展示。
3. 目标是提高首屏可见条目数，而不是维持大卡片式留白。

### 2. 零值降噪

以下列在主值为 `0` 时必须降噪：

1. `Token统计`
2. `请求统计`

降噪规则：

1. 仅显示主值 `0`。
2. 不再显示第二行的解释性文字。

### 3. 横向空间优先级

在列宽分配上，优先级从高到低如下：

1. 站点列
2. 余额列
3. 今日消费 / Token统计 / 请求统计
4. 模型数 / CLI可用性
5. 右侧操作区

说明：

1. 尽管本次选择的是“保留更多列同时可见”的方向，列宽仍需优先保证站点识别和金额可读性。
2. 右侧动作区允许压缩，但不允许牺牲主信息列来换取过宽的动作留白。

## 行为保持与兼容要求

1. 现有展开区交互必须保持可用：
   - 展开。
   - 收起。
   - 动画过渡。
   - 展开后的详情承载。
2. 现有跨天归零逻辑不改。
3. 现有排序逻辑保持以下约束：
   - `Token统计` 只按总 Token 排序。
   - `请求统计` 不支持排序。
   - `CLI可用性` 不提供排序。
4. `LDC比例` 的后端字段与数据链路不动，仅在前端移除显示。

## 预期影响文件

本次设计预期主要影响以下前端文件：

1. `src/renderer/pages/SitesPage.tsx`
2. `src/renderer/components/SiteListHeader/SiteListHeader.tsx`
3. `src/renderer/components/SiteCard/SiteCardHeader.tsx`
4. `src/renderer/components/SiteCard/SiteCardActions.tsx`
5. 与列宽、样式、测试相关的配套文件

不预期修改：

1. 后端处理器
2. 持久化结构
3. 展开区主体组件的承载模型

## 验证要求

### 1. UI 结果验证

至少验证以下场景：

1. 常规桌面窗口宽度下无横向滚动条。
2. 站点列第一行显示站点名，第二行左侧显示账户名，第二行右侧显示更新时间。
3. `LDC比例` 在前端不再显示。
4. `CLI可用性` 列标题正确显示。
5. `CLI配置` 作为显性文字入口存在并可用。
6. `Token统计 / 请求统计` 在零值时只显示 `0`。
7. 余额列在 `100000` 阈值两侧分别按完整值与缩写值展示。
8. 展开区仍可正常展开与收起。

### 2. 测试策略

以 Renderer 层测试为主：

1. 优先更新站点页相关渲染测试。
2. 必要时补以下断言：
   - 列标题变更。
   - `LDC比例` 隐藏。
   - 更新时间并入站点列。
   - `CLI配置` 文字入口。
   - 零值降噪。
   - 余额阈值格式化。

## 实施约束

1. 本次改版不得顺带改动展开区详情内容。
2. 不得把当前卡片展开模型重写为全新表格详情模型。
3. 不得修改后端数据结构来配合前端布局。
4. 不得在未验证默认桌面宽度表现之前继续增加新列或新增显性文字按钮。
