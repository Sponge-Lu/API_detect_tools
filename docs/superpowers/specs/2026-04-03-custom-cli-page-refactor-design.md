# 自定义 CLI Tab 页结构重构设计

日期：2026-04-03

## 1. 背景

当前自定义 CLI 页面实现位于 `src/renderer/pages/CustomCliPage.tsx`，整体仍是“卡片式配置列表 + 弹窗编辑器”的结构。右侧编辑能力主要承载在 `src/renderer/components/dialogs/CustomCliConfigEditorDialog.tsx` 中。

现状与目标存在这些结构性偏差：

- 左侧是卡片列表，不是配置表。
- 右侧编辑能力依赖弹窗，不是常驻工作区。
- 配置预览常驻展示，信息密度过高。
- 左侧点击切换交互不符合“整行选中”的预期。
- CLI 测试展示与当前用户想要的三列工作区结构不一致。

本次重构只处理页面结构、交互组织和已有能力的重排，不引入新的业务概念，不改变现有存储模型的核心字段。

## 2. 目标

- 将页面重构为固定左右分栏。
- 左侧改为“配置表”，右侧改为“配置编辑”。
- 保留现有配置编辑、模型拉取、CLI 预览编辑、测试模型选择、CLI 测试结果等能力。
- 左侧 `CLI测试` 三图标与站点管理保持完全相同的语义、样式层级和 tooltip 说明。
- 删除常驻预览与 CLI 映射区域，改为更聚焦的工作流。

## 3. 非目标

- 不改动 `CustomCliConfig` 的核心数据结构设计。
- 不重做 CLI 配置生成逻辑。
- 不改动站点管理页的 CLI 图标规则。
- 不改变“每个 CLI 最多选择 3 个测试模型”的业务限制。

## 4. 已确认约束

- 页面骨架采用“方案 A”：左侧配置表为主导航，右侧为常驻编辑工作区。
- 左侧允许纵向滚动，但不显示垂直滚动条。
- 左侧每一行配置不使用圆角卡片外框，采用扁平表格行。
- 进入 tab 后如果存在配置，默认自动选中第一条并显示右侧编辑区。
- 左侧点击某条配置的整行任意位置都可以切换选中项。
- 当前选中行必须用更深背景色明确表达。
- 左侧 `CLI测试` 列必须与站点管理中的三图标表达完全一致，包括：
  - 未启用
  - 未配置
  - 已配置待测试
  - 测试失败
  - 测试通过
- 左侧三图标必须保留与站点管理一致的 tooltip，包括详细状态文本与测试时间。
- 右侧测试区必须重构为三列结构，每个 CLI 各自有三个测试模型选择框和各自的测试按钮。
- 点击某个 CLI 的测试按钮时，只测试该 CLI 当前选中的最多 3 个模型，不触发其他 CLI。

## 5. 当前实现证据

### 5.1 页面入口

`src/renderer/pages/CustomCliPage.tsx` 当前直接渲染：

- 顶部工具栏
- 左侧配置“卡片列表”
- `ApplyCliPopover`
- `CustomCliConfigEditorDialog`

该页面当前并不存在真正的左右工作区。

### 5.2 编辑器能力来源

`src/renderer/components/dialogs/CustomCliConfigEditorDialog.tsx` 当前已具备：

- 基础字段编辑
- 模型拉取
- 每个 CLI 的启用开关与模型选择
- 每个 CLI 的测试模型选择，且最多 3 个
- 配置预览生成
- 预览内容编辑、保存、取消、重置
- 按 CLI 分别缓存 `editedFiles`
- CLI 测试结果记录

这意味着本次重构应优先复用这些成熟行为，而不是重写业务逻辑。

### 5.3 左侧图标语义来源

`src/renderer/components/CliCompatibilityIcons/CliCompatibilityIcons.tsx` 已明确定义：

- 未启用：`opacity-15 grayscale`
- 未配置：`opacity-25 grayscale`
- 已配置待测试：`opacity-50 grayscale`
- 测试失败：`opacity-70 grayscale brightness-75`
- 测试通过：`opacity-100`

同时 tooltip 文案由 `getStatusText`、`getCodexDetailText`、`getGeminiDetailText` 与测试时间格式化共同组成。本次左侧 `CLI测试` 列必须复用这套规则，而不是重写一套近似规则。

## 6. 设计方案

### 6.1 页面总布局

页面改为左右分栏：

- 左侧：`配置表`
- 右侧：`配置编辑`

顶层不再是单列滚动卡片区。左右两栏作为同一工作台长期并存。

### 6.2 左侧“配置表”

左侧标题改为“配置表”，不显示说明文字。

左侧展示固定五列：

- 名称
- BaseURL
- 模型数
- CLI测试
- 备注

行为规则：

- 每条配置渲染为扁平表格行。
- 不再使用卡片圆角外框。
- 行的任何区域都可点击选中。
- 当前选中行使用更深背景色高亮。
- 左侧内容超出时允许滚动，但隐藏垂直滚动条。

列说明：

- `名称`：显示配置名。
- `BaseURL`：显示基础地址，保留点击打开与复制能力。
- `模型数`：显示模型数量。
- `CLI测试`：显示 3 个 CLI 图标，按站点管理同规则渲染。
- `备注`：显示摘要文本，超出截断。

### 6.3 左侧 `CLI测试` 列

该列不显示模型名或配置按钮，只显示三枚图标：

- Claude Code
- Codex
- Gemini CLI

图标状态由该配置项当前保存的配置与测试结果共同决定。语义、透明度、灰度、tooltip 文案、Codex/Gemini 的详细状态补充、测试时间文案，全部与站点管理保持一致。

实现上应优先复用 `CliCompatibilityIcons` 的状态计算逻辑，必要时提取共享函数，而不是复制一套不同实现。

### 6.4 右侧“配置编辑”

右侧标题改为“配置编辑”，不显示说明文字。

右侧直接承接当前编辑器中的核心内容：

- 配置名称
- Base URL
- API Key
- 备注
- 模型拉取与模型数量状态
- 各 CLI 的启用状态
- 各 CLI 的主模型选择

右侧不再保留当前“配置预览”整块常驻区域，也不显示 CLI 映射。

### 6.5 每个 CLI 的编辑行

每个 CLI 在右侧配置区中都应有自己的编辑行或编辑块，至少包括：

- CLI 图标与名称
- 启用开关
- 主模型选择器
- `预览` 按钮

`预览` 按钮点击后，打开该 CLI 的配置预览弹层。

### 6.6 预览弹层

预览弹层按单个 CLI 工作：

- 只展示当前 CLI 生成出的配置文件
- 支持 `编辑`
- 支持 `保存`
- 支持 `取消`
- 支持 `重置`

状态规则：

- 每个 CLI 继续单独维护自己的已编辑预览缓存。
- 切换不同 CLI 时，彼此的已编辑内容不能互相覆盖。
- 当基础字段变化时，默认预览内容应按新字段重新生成。
- 如果某个 CLI 已存在手工编辑缓存，则优先显示缓存，直到用户主动重置。
- `重置` 只重置该 CLI 的预览编辑内容，不重置整条配置的基础字段。

保存整条配置时，应继续将各 CLI 的 `editedFiles` 一并保存，保持与现有数据结构兼容。

### 6.7 CLI 测试区

右侧单独保留一个 `CLI 测试` 区域，不沿用当前“结果卡片列表”结构。

新结构如下：

- 横向排列 3 个 CLI 列
- 每列对应一个 CLI
- 每列下方竖向排列 3 个测试模型选择框
- 每个选择框后方直接显示该模型对应的测试结果
- 每个 CLI 列底部各自有一个测试按钮

交互规则：

- 每个 CLI 最多选择 3 个测试模型，延续现有限制。
- 测试按钮只测试当前 CLI。
- 点击某 CLI 的测试按钮后，只依次测试该 CLI 当前已选的最多 3 个模型。
- 不触发另外两个 CLI。
- 测试结果写回该 CLI 自己的结果槽位，并用于更新左侧三图标的状态。

### 6.8 默认态与切换态

进入页面时：

- 如果存在配置，默认自动选中第一条。
- 右侧立即显示第一条配置的编辑内容。

切换配置时：

- 左侧选中态立即变化。
- 右侧所有基础字段、CLI 设置、测试模型、预览缓存与测试结果上下文都切换到对应配置。

## 7. 实现策略

### 7.1 复用优先

优先复用现有逻辑：

- `useCustomCliConfigStore`
- `generateClaudeCodeConfig`
- `generateCodexConfig`
- `generateGeminiCliConfig`
- 当前编辑器中的 `perCliEdited`
- 当前编辑器中的测试模型与测试结果逻辑

目标是“重组 UI 和交互”，而不是“重写业务逻辑”。

### 7.2 组件调整方向

计划中的主要调整点：

- 在 `CustomCliPage.tsx` 中重建页面布局与选中态。
- 将 `CustomCliConfigEditorDialog.tsx` 中可复用的编辑区逻辑抽离为可嵌入右侧工作区的组件，或直接重构该组件使其同时服务于页面内编辑。
- 将常驻预览改为由每个 CLI 的 `预览` 按钮触发的弹层。
- 为左侧 `CLI测试` 列接入与站点管理一致的状态/tooltip 规则。

### 7.3 数据兼容

保持以下字段继续有效：

- `cliSettings[cli].model`
- `cliSettings[cli].enabled`
- `cliSettings[cli].testModels`
- `cliSettings[cli].editedFiles`

这样可以降低迁移风险，并保证已有配置不需要额外转换。

## 8. 风险与处理

### 8.1 预览逻辑与页面编辑区分离风险

当前预览、编辑、重置逻辑集中在弹窗组件里。若直接拆分，容易破坏按 CLI 缓存编辑内容的行为。

处理方式：

- 优先抽离状态逻辑，再调整展示层。
- 在重构前先补回归测试覆盖预览编辑缓存行为。

### 8.2 左侧图标语义偏差风险

如果左侧 `CLI测试` 自己实现状态映射，极易与站点管理出现细微不一致。

处理方式：

- 复用现有状态计算函数或提取共享逻辑。
- 测试 tooltip 与状态类名映射。

### 8.3 测试区重构后的行为回归风险

当前测试逻辑偏向“聚合式执行”。改成“每个 CLI 单独测试”后，容易误触发其他 CLI。

处理方式：

- 为单 CLI 测试按钮单独补测试。
- 明确验证“只触发自身，不触发其他 CLI”。

## 9. 测试策略

至少覆盖以下行为：

- 进入页面默认选中第一条配置。
- 点击左侧整行任意位置可以切换右侧编辑对象。
- 左侧当前选中行高亮。
- 左侧表格渲染固定五列。
- 左侧不再使用卡片式圆角列表样式。
- 左侧 `CLI测试` 图标状态与 tooltip 符合站点管理规则。
- 右侧不再常驻显示配置预览。
- 每个 CLI 行存在 `预览` 按钮。
- 预览弹层支持编辑、保存、取消、重置。
- `CLI 映射` 区域不再显示。
- CLI 测试区按三列布局渲染。
- 每个 CLI 只有自己的测试按钮。
- 点击某 CLI 测试按钮时，只测试该 CLI 当前选中的最多 3 个模型。
- 测试结果会更新该 CLI 的显示结果，并反馈到左侧图标状态。

## 10. 验收标准

满足以下条件即可认为本次设计落地成功：

- 页面从“卡片列表 + 弹窗编辑”转为“左表右编”的常驻工作区。
- 左侧五列完整、可整行选中、选中态明确。
- 左侧三图标与站点管理完全同语义、同 tooltip。
- 右侧基础编辑区承接原有弹窗编辑能力。
- 常驻预览消失，改为按 CLI 按钮触发预览弹层。
- CLI 映射消失。
- 测试区改为三列独立结构。
- 每个 CLI 只测试自己。
- 现有配置数据可直接继续使用，无需迁移。
