# brainstorm: simplify route model redirection

## Goal

把当前“模型重定向”从“先对所有模型做自动归类，再裁成 3 个展示”收敛成更简单的流程：每个厂商只自动挑选 3 个最先进模型生成默认重定向；其他模型不做自动归类，只作为用户手工编辑/新增时的原始来源候选。同时把真实选路从“统计分数优先的通道重试”明确改成“重定向模型 -> 原始模型 -> 具备可用用户分组与 API key 的站点 -> 站点优先级 -> API key 优先级”的确定性路径。

## What I already know

* 当前主进程在 `src/main/route-model-registry-service.ts` 中会遍历所有站点/账户的 `cached_data.models`，并对每个原始模型都调用 `processModelSource(...)` 建立 `registry.entries`。
* 当前 `displayItems` 的 seeded 部分虽然只取每个厂商排序后的前 3 个条目，但这一步发生在“所有模型都已经被自动归类进 entries”之后。
* 当前前端 `buildVendorSections()` 里的 `candidateGroups` 来自 `vendorEntries`，也就是来自整包 `registry.entries`，因此编辑弹窗实际上是建立在“全量自动归类后的结果”上。
* 当前代理层在 `src/main/route-proxy-service.ts` 中，对未命中 `registry.entries` 的模型会直接回退到 raw model 透传，因此“非前三模型不自动归类”在转发链路上是可成立的。
* 用户最新纠正：不需要把所有模型都做自动归类；每个厂商只要挑 3 个最先进的模型做模型重定向即可。
* 用户此前已经明确：这 3 个自动项只在初始化/重建目录时生成；之后用户修改它们时，应直接覆盖，不再重新自动归类。
* 用户此前已经明确：仍然需要支持用户编辑“重定向名称 / 原始名称”，并新增额外的重定向模型与规则。
* 当前编辑弹窗已经改成“重定向名称 + 全站点模型列表”，原始模型列表不再按厂商过滤。
* 当前 `ModelRedirectionTab` 的来源详情是按站点/账户聚合 `entry.sources`，直接展示来源数、原始模型列表、用户分组和 API key 名称，还没有按“某个原始模型实际可用的用户分组”做过滤。
* 当前真实请求链路是：`route-proxy-service.ts` 先匹配规则，再调用 `resolveChannels(rule, canonicalModel)` 展平所有可用站点/账户/API key，最后由 `sortChannelsByScore(...)` 按运行时统计分数排序并依次重试。
* 当前没有独立的“站点优先级 / API key 优先级”持久化字段；`route-channel-resolver.ts` 也没有基于重定向来源、用户分组或 API key 优先级做过滤与排序。

## Assumptions (temporary)

* “不做所有模型自动归类”指的是：非前三模型不自动生成 canonical 映射/默认重定向规则，但仍可在编辑器中作为原始来源被搜索并勾选。
* 用户新增的手工重定向模型仍然需要创建可见卡片；它们不属于自动挑选的 3 个默认模型。
* 显式点击“重建目录”时，允许重新按最新模型列表生成每个厂商的 3 个默认重定向项；平时编辑/删除/新增都不应触发全量自动归类。
* “有用户分组可以使用这个原始模型并且创建了 apikey 的站点”指的是：某站点/账户下，某个原始模型对应的用户分组中，至少有一个分组存在已创建的 API key，才算这个站点对该原始模型可用。
* 新增的优先级配置需要参与真实请求选路，不只是 UI 展示。
* 每个 CLI 在路由配置中始终只选择一个“重定向后的模型”，因此选路回退范围应当限定在这个重定向模型可达的站点/API key 集合之内。

## Requirements (evolving)

* 初始化/重建目录时，每个厂商只自动挑选 3 个最先进模型生成默认模型重定向。
* 非前三模型不应被自动归类成 canonical entry，也不应自动生成默认重定向规则。
* 非前三模型仍应作为“原始名称”候选出现在编辑/新增弹窗里，支持搜索、勾选和多选。
* 用户可以编辑这 3 个默认重定向项，编辑后直接覆盖对应规则，不再触发额外自动归类。
* 用户编辑默认重定向项时，只覆盖该默认槽位本身，不额外生成新的手工项。
* 用户可以删除默认/手工重定向项；删除后不应自动补出新的默认项，除非用户显式执行“重建目录”。
* 用户可以新增手工重定向模型与规则；新增后必须有可见卡片。
* 路由页默认仍按每个厂商单列折叠展示，头部摘要只展示当前默认的 3 个推荐重定向名称。
* 厂商展开后显示“默认 3 项 + 所有手工新增项”。
* CLI 选择器展示的仍应是“当前可见的重定向后模型名称”，而不是原始模型。
* 不能破坏现有 sourceKey 级 override 保存逻辑。
* 未命中任何重定向规则的模型，在代理层应继续 raw 透传。
* 编辑弹窗中的每个原始模型后面可以显示 `站点数 / 来源数`。
* 原始模型列表不再显示“是否已重定向 / 是否已有手工覆盖”之类的标记。
* 来源详情需要按“当前重定向模型 -> 原始模型 -> 站点”来过滤，只显示该原始模型对应的可用用户分组。
* 每个厂商卡片需要增加“站点优先级”按钮，打开优先级弹窗。
* 优先级弹窗中需要显示站点优先级，默认值为 `10`；数字越小优先级越高。
* 优先级弹窗中还需要显示站点内 API key 优先级，默认值为 `3`；数字越小优先级越高。
* 优先级弹窗里只显示“当前厂商下多个重定向规则中选中的原始模型所对应的 API key”；无关 API key 不显示。
* 如果同一个站点下存在多个 active 账户，优先级弹窗里的 API key 按站点合并展示，但每个 API key 要显示来源账户。
* 真实请求链路需要调整为：`重定向模型 -> 原始模型 -> 有可用用户分组且已创建 API key 的站点 -> 站点优先级 -> API key 优先级 -> 请求上游`。
* 在无显式优先级配置时，新的优先级排序应优先于当前的统计分数排序；统计分数可以作为更后面的 fallback tie-breaker。
* 站点优先级和 API key 优先级按“厂商共享”配置，不按单个重定向模型拆分。
* 每个 CLI 的真实请求回退范围，仅限于该 CLI 当前选中的“重定向后模型”能够连接到的站点/API key。
* 当首选组合失败时，回退顺序采用：先同站点下一个 API key，再切到下一个站点。
* 真实选路时，应先根据站点优先级和 API key 优先级确定当前使用的站点/API key 组合；只有在该组合下存在多个原始模型时，才按用户在编辑弹窗里的勾选顺序选择原始模型。

## Technical Approach

把“自动默认重定向”和“原始来源候选池”拆成两层：

* 自动层：只负责每个厂商 3 个 seeded display item/default redirect。
* 手工层：允许用户从全部站点检测到的原始模型中搜索并勾选来源，创建或覆盖手工规则。
* 代理解析层：仅对已有重定向规则的模型做 canonical 解析；其余模型维持 raw 透传。
* 选路层：在规则命中之后，不再直接把所有候选通道交给统计分数排序，而是先根据“CLI 当前选中的重定向模型 / 站点优先级 / API key 优先级”确定候选组合，再在该组合内按用户勾选顺序解析原始模型，并按“站点内 key 回退优先于跨站点回退”的策略执行。

## Decision (ADR-lite)

**Context**: 当前实现先把全部原始模型归类进 `registry.entries`，再从中裁出 3 个展示。这与“只自动重定向 3 个模型”的产品规则不一致，也让编辑器、展示和代理语义都依赖“全量自动归类”。

**Decision**: 调整为“只对每个厂商的 3 个默认模型做自动重定向”；其余模型只保留为手工编辑时可选的原始来源，不再默认进入自动重定向目录。并新增厂商级站点/API key 优先级配置，驱动真实请求选路。

**Consequences**:

* 优点：产品语义更直接，编辑/删除/新增的行为边界清晰。
* 优点：路由展示、CLI 选择器和代理解析都能围绕“当前真正存在的重定向规则”工作。
* 代价：需要把当前依赖 `registry.entries` 的编辑候选来源逻辑拆开，改为基于原始来源池而不是“全量自动归类结果”。
* 代价：需要在 `route-channel-resolver.ts` / `route-proxy-service.ts` 中引入新的一层确定性候选过滤与排序，替换当前“直接按 stats 分数重试”的默认路径。
* 已确认：非前三模型保留为手工编辑/新增时的原始来源候选，不进入自动默认重定向目录。
* 已确认：展开区显示“默认 3 项 + 手工新增项”；编辑默认项只覆盖原槽位，不额外派生新手工项。
* 已确认：编辑弹窗只保留“重定向名称 + 全模型列表”，原始模型旁可显示 `站点数 / 来源数`。
* 已确认：站点优先级和 API key 优先级按厂商共享配置。
* 已确认：回退只发生在当前 CLI 所选重定向模型的可达站点/API key 集合内，且先同站点换 key，再跨站点切换。
* 已确认：同站点下多个账户的 API key 在优先级弹窗里按站点合并展示，并显示来源账户。
* 已确认：如果某个站点/API key 组合下可用原始模型不止一个，则按用户在编辑弹窗中的勾选顺序选择原始模型。

* 代码证据：
  * `route-model-registry-service.ts` 中 189-223 行会遍历所有 detected models 并调用 `processModelSource(...)`。
  * 同文件 226-230 行表明当前 `registry.entries` 是全量扫描结果，`displayItems` 只是基于这些 entries 再裁切。
  * `ModelRedirectionTab.tsx` 中当前来源详情是按 `entry.sources` 聚合站点与用户分组展示，还没有按“某个原始模型对应的可用用户分组”过滤。
  * `route-channel-resolver.ts` 中 134-205 行表明当前会把满足规则的站点/账户/API key 全部展平成候选通道，没有站点/API key 优先级字段。
  * 当前真实通道粒度是 `siteId + accountId + apiKeyId`，不是单纯的 `siteId + apiKeyId`；新的优先级 UI 需要把 account 细节折叠到“站点内 API key 列表”里。
  * `route-proxy-service.ts` 中 253-323 行表明当前候选通道会先交给 `sortChannelsByScore(...)`，再逐个尝试。
  * `route-stats-service.ts` 中 113-147 行表明当前排序依据是成功率/失败惩罚/最近成功/延迟，而不是显式优先级。
  * `route-proxy-service.ts` 中 221-231 行表明：若 `registry.entries` 无匹配，模型会直接回退到 raw 透传。
* 当前实现与新要求的主要冲突：
  * 自动层和手工层都绑定在 `registry.entries` 上，导致“全量自动归类”与“编辑候选来源池”耦合。
  * 需要把“默认 3 项”与“手工新增项”作为展示层的两类对象来维护。
  * 需要让 seeded item 在被编辑后仍保留原 display item 身份，而不是通过新增 manual item 的方式体现。
  * 当前请求链路没有“重定向模型 -> 原始模型 -> 具备可用用户分组和已建 key 的站点 -> 优先级”这层中间选择器。
  * 当前数据模型里没有站点优先级/API key 优先级存储位置，需要新增 shared contract 与 config persistence。
  * 当前 CLI 选路只基于 `canonicalModel` + 规则过滤后的候选通道，没有把“当前 CLI 选择了哪个重定向模型”作为回退边界显式固化。
  * 当前 display item 只保存 `sourceKeys` 集合，没有显式记录“用户勾选顺序”，而新的原始模型选择规则依赖这个顺序，可能需要补充顺序字段或保序来源列表。
